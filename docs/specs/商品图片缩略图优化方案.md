# 商品图片缩略图优化方案

> 日期：2026-06-29
> 状态：待实施
> 版本：v4（Review 反馈修正 6 项遗漏）

---

## 1. 背景与问题

### 1.1 业务痛点

平台所有商品图片（列表卡片、详情缩略导航、购物车、最近浏览、运营列表）均加载原图。
列表页一次 20 张卡片，OVH 服务器带宽有限，用户体感加载慢。

### 1.2 数据规模

| 指标 | 当前 | 近期预期 |
|------|------|----------|
| 商品数 | ~1,000 | 10,000+ |
| 图片总数 | ~1,600 | 50,000-80,000（平均 5-8 张/品） |
| 存储总量 | ~150MB | **10GB+** |
| 格式分布 | JPEG 1,292 + PNG 292 + JPEG(扩展名) 5 | — |

### 1.3 根因分析

**两个独立问题叠加：**

**问题 A — 没有缩略图机制：**
所有场景统一用原图，56px 的缩略图条也加载 800×800 JPEG。

**问题 B — XFS 导入图片未经压缩：**
手动上传经过 `_prepare_image()` 压缩到 800×800 JPEG q85，但 XFS 导入的 `_copy_image()` **原样复制，不做任何处理**，导致大量 1-3MB 原图直接进库。

## 2. 现状分析

### 2.1 图片存储结构

```
uploads/products/{product_id}/{uuid}.jpg      ← 手动上传（经压缩，~40-80KB）
uploads/products/{spu_code}/{filename}.jpg     ← XFS 导入（原样复制，500KB-3MB）
uploads/products/{spu_code}/{filename}.png     ← XFS 导入 PNG（原样复制）
```

### 2.2 两个图片入口的处理差异

| 入口 | 处理逻辑 | 输出大小 |
|------|----------|----------|
| 手动上传 `save_uploaded_image()` | 缩放 800×800 + 正方形白底 + JPEG q85 | 40-80KB |
| XFS 导入 `_copy_image()` | **原样复制，不处理** | 500KB-3MB |

### 2.3 后端 API 图片返回方式——4 个独立出口

| 出口 | 函数 | 返回值 | 文件位置 |
|------|------|--------|----------|
| 买方列表 | `_batch_main_images()` | `dict[int, str]`（product_id → **完整 URL**） | `services/product.py:473` |
| 买方详情 | `_img_to_dict()` | `full_url` 字段 | `api/v1/products.py:168` |
| 运营列表 | 同 `_batch_main_images()` | 同上 | `api/v1/operator_products.py:224` |
| 最近浏览 | `_get_product_main_image()` | 单个 URL | `services/buyer_event.py:203` |
| 购物车 | `_resolve_main_image_from_product()` | 单个 URL | `services/cart.py:560` |

**关键：`_batch_main_images()` 返回的是拼好的完整 URL（`/static/{key}`），不是原始 `image_key`。** 要在 API 层从 URL 反推 key 不可靠。

### 2.4 后端 Schema 涉及的类型

| Schema | `main_image` 字段 | 文件位置 |
|--------|-------------------|----------|
| `ProductPublic` | `main_image: str \| None` | `schemas/product.py:257` |
| `ProductOperator` | `main_image: str \| None` | `schemas/product.py:307` |
| `ProductImageSchema` | `full_url: str` | `schemas/product.py:20` |
| `CartItemPublic` | `main_image: str \| None` | `schemas/cart.py:58` |

### 2.5 前端图片使用

| 场景 | 组件 | 显示尺寸 | 加载的图 |
|------|------|----------|---------|
| 列表页 ProductCard | `ProductCard.tsx` | ~200×200 | 原图 |
| 楼层卡 ProductCardCompact | `ProductCardCompact.tsx` | ~160×160 | 原图 |
| 最近浏览 | `RecentViews.tsx` | 112×112 | 原图 |
| 购物车 | `cart/page.tsx` | 80×80 | 原图 |
| 运营列表 | `operator/products/page.tsx` | 44×44 | 原图 |
| 详情缩略图条 | `ProductGallery.tsx` | 56×56 | 原图 |
| 详情主图 | `ProductGallery.tsx` | 400×360 | 原图（合理） |
| Lightbox 全屏 | `ProductGallery.tsx` | 视口级 | 原图（合理） |

### 2.6 缺失的行业标配优化

| 优化项 | Shopify/Alibaba 做法 | 当前状态 |
|--------|---------------------|---------|
| 多尺寸缩略图 | 上传时生成 2-3 档 | 无 |
| WebP 格式 | 默认输出 WebP | 全 JPEG/PNG |
| `loading="lazy"` | 全部图片 | 部分有 |
| `decoding="async"` | 全部图片 | 全部缺 |
| HTTP 缓存头 | `immutable` + 长 `max-age` | nginx 7 天，FastAPI 无设置 |
| 图片尺寸提示 | `width`/`height` 属性防 CLS | 全部缺 |

### 2.7 静态文件缓存现状

```nginx
# deploy/nginx.conf — 当前配置
location /static/ {
    proxy_pass http://backend;
    expires 7d;
    add_header Cache-Control "public";
}
```

nginx 层已有 7 天缓存，但不是 immutable。FastAPI 层无缓存头。

## 3. 设计方案

### 3.1 核心思路

**上传/导入通过统一入口保存原图并同步生成 WebP 缩略图；后端所有图片出口统一返回缩略图 URL；前端按场景选用；部署层加长缓存。**

### 3.2 关键设计决策

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 缩略图格式 | **WebP** | 比 JPEG 小 25-35%，现代浏览器全支持（东非用户主力 Chrome/Android） |
| 缩略图尺寸 | **一档 300×300** | 覆盖列表(200px)/购物车(80px)/缩略图条(56px)全部中小场景 |
| 是否加 DB 字段 | **不加** | 缩略图路径由 `image_key` 约定推导，确定性映射 |
| API 返回缩略图字段 | **加** | 后端推导比前端推导更可靠，换存储方案时只改后端 |
| 生成时机 | **上传时同步** | PIL resize 300px <50ms，无需异步队列 |
| 存量原图处理 | **仅 `.jpg/.jpeg` 超尺寸的压缩；`.png/.webp` 不动** | 避免扩展名与实际格式不一致 |
| HTTP 缓存 | **nginx 层改 immutable + 1 年** | 文件名含 uuid，内容不变 |

### 3.3 缩略图命名规则（约定推导）

```
原图:     products/529/a1b2c3d4.jpg
缩略图:   products/529/a1b2c3d4_thumb.webp

原图:     products/P-XFS-123/main_01.png
缩略图:   products/P-XFS-123/main_01_thumb.webp
```

### 3.4 统一工具函数 `thumb_key_from_image_key()`

**后端所有需要缩略图路径的地方统一调用这一个函数**，不各自推导：

```python
# 放在 _buyer_utils.py 或独立 image_utils.py
def thumb_key_from_image_key(image_key: str) -> str:
    """image_key → 缩略图 key（约定推导，不查 DB）。"""
    stem, _ = os.path.splitext(image_key)
    return f"{stem}_thumb.webp"

def thumb_url_from_image_key(image_key: str) -> str:
    """image_key → 缩略图完整 URL。"""
    return f"{settings.IMAGE_PATH_PREFIX}/{thumb_key_from_image_key(image_key)}"
```

### 3.5 缩略图参数

| 参数 | 值 | 说明 |
|------|-----|------|
| 最大边 | 300px | `thumbnail()` 等比缩放 |
| 格式 | WebP | `img.save(format="WEBP")` |
| 质量 | 80 | 视觉无损 |
| 预估体积 | 8-15KB/张 | 原图 40KB-3MB → 缩略图 8-15KB |

### 3.6 场景用图规则

| 场景 | 原图/缩略图 | 原因 |
|------|------------|------|
| 列表页卡片 | **缩略图** | 显示 ≤200px |
| 楼层卡片 | **缩略图** | 显示 ~160px |
| 最近浏览 | **缩略图** | 显示 112px |
| 购物车 | **缩略图** | 显示 80px |
| 运营列表 | **缩略图** | 显示 44px |
| 详情缩略图条 | **缩略图** | 显示 56px |
| 详情主图 + 放大镜 | **原图** | hover 放大需要高分辨率 |
| Lightbox 全屏 | **原图** | 全屏查看需要原始画质 |
| Lightbox 底部缩略图 | **缩略图** | 显示 48px |

### 3.7 降级机制

**策略：前端 `<img onError>` 降级到原图。**

```tsx
<img
  src={imageUrl(item.main_image_thumbnail)}
  onError={(e) => {
    const target = e.currentTarget;
    if (target.dataset.fallback) return; // 防止循环
    target.dataset.fallback = "1";
    target.src = imageUrl(item.main_image);
  }}
  loading="lazy"
  decoding="async"
/>
```

后端始终返回推导的缩略图 URL（不校验文件是否存在，避免 IO 开销）。如果缩略图文件还没生成（存量数据补生之前）或生成失败，前端 `onError` 自动切回原图。

**部署顺序保障：** 推荐先跑完存量脚本再部署前端新代码，这样 `onError` 只是兜底而非常态。

**文件：** `backend/app/services/_buyer_utils.py`（或独立 `image_utils.py`）

```python
THUMB_SIZE = (300, 300)
THUMB_WEBP_QUALITY = 80

def thumb_key_from_image_key(image_key: str) -> str:
    """image_key → 缩略图 key。"""
    stem, _ = os.path.splitext(image_key)
    return f"{stem}_thumb.webp"

def thumb_url_from_image_key(image_key: str) -> str:
    """image_key → 缩略图完整 URL。"""
    return f"{settings.IMAGE_PATH_PREFIX}/{thumb_key_from_image_key(image_key)}"

def generate_thumbnail(original_path: Path) -> Path | None:
    """为原图生成 WebP 缩略图。幂等：已存在则跳过。"""
    thumb_path = original_path.with_name(
        original_path.stem + "_thumb.webp"
    )
    if thumb_path.exists():
        return thumb_path
    try:
        img = Image.open(original_path).convert("RGB")
        img.thumbnail(THUMB_SIZE, Image.LANCZOS)
        img.save(thumb_path, format="WEBP", quality=THUMB_WEBP_QUALITY)
        return thumb_path
    except Exception:
        logger.warning("缩略图生成失败: %s", original_path, exc_info=True)
        return None
```

### 4.3 存量数据处理

**文件：** `backend/scripts/generate_thumbnails.py`（新文件）

#### 两个任务

| 任务 | 对象 | 操作 |
|------|------|------|
| A. 压缩超尺寸原图 | `.jpg/.jpeg` 文件且尺寸 >800px | 重压缩到 800×800 JPEG q85，覆盖原文件 |
| B. 生成缩略图 | 所有图片文件 | 生成 `_thumb.webp` |

**重要：`.png` 和 `.webp` 原图不做压缩覆盖**，避免扩展名与实际格式不一致。只生成 WebP 缩略图。

#### 设计要求

| 要求 | 实现 |
|------|------|
| 幂等 | 缩略图已存在 → 跳过 |
| 并发 | `ProcessPoolExecutor`，默认 CPU 核数 |
| 断点续跑 | 中断后重跑自动跳过已处理 |
| dry-run | `--dry-run` 只统计不执行 |
| 进度可视 | tqdm 进度条 |
| 错误隔离 | 单张失败不中断，记录到 stderr |
| 完成统计 | 输出处理/跳过/失败数量、节省空间、耗时 |

#### 用法

```bash
# 预览
python scripts/generate_thumbnails.py --dry-run

# 执行
python scripts/generate_thumbnails.py --workers 4

# 单目录
python scripts/generate_thumbnails.py --product-dir products/P-XFS-20740095
```

## 6. 效果预估

### 6.1 流量对比

| 场景 | 优化前 | 优化后 | 降幅 |
|------|--------|--------|------|
| 列表页单张（手动上传图） | ~60KB | ~12KB | 80% |
| 列表页单张（XFS 导入图） | 500KB-3MB | ~12KB | 97-99% |
| 列表页 20 张总下载 | 1-40MB | ~**240KB** | 90-99% |
| 二次访问同一页 | 重新下载 | **0**（immutable 缓存） | 100% |

### 6.2 用户体感

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| 列表页首屏（OVH） | 3-8s | <1s |
| 翻页/回退 | 重新加载 | 瞬间（缓存命中） |
| CLS 布局偏移 | 有 | 无（width/height 属性） |

### 6.3 存储影响

| 指标 | 变化 |
|------|------|
| 缩略图额外占用 | +500MB-1GB（5-8 万张 × 10-15KB） |
| JPG 原图压缩节省 | -3-5GB |
| **净效果** | **磁盘占用减少** |

| 多尺寸缩略图 | 品类超 5 万、需要极致优化 | 扩展 `_medium.webp` / `_large.webp` 档位 |
| CDN 加速 | 用户量增长、多地域访问 | 对象存储 + CDN，`public_url()` 返回 CDN 域名 |
| 响应式 srcset | 移动端优化需求 | `<img srcset="...">` 不同设备加载不同尺寸 |
| Nginx try_files 降级 | 去除前端 onError 逻辑 | `try_files $uri @fallback_original` |
