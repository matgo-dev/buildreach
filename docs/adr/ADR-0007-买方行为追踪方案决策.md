# ADR-0007: 买方行为追踪方案决策

**状态**: 已批准  
**日期**: 2026-06-17  
**决策者**: liujingjing

---

## 1. 背景

平台需要追踪买方在站内的操作路径（浏览商品、搜索、加购、提询价等），用途两层：

1. **买方可见** — "最近浏览/足迹"功能，电商标配，提升复访转化
2. **运营分析** — 操作路径追踪、转化漏斗、热门商品统计，辅助运营决策

现有 `audit_logs` 表只记写操作（POST/PUT/DELETE），且定位是安全审计，不适合承载高频读事件。

## 2. 行业参考：阿里国际站

### 2.1 阿里的买家行为追踪能力

| 层级 | 能力 | 技术手段 |
|---|---|---|
| **L1 事件采集** | 全链路前后端埋点，覆盖搜索/浏览/询盘/TM/成交 | 前端 JS SDK + 后端 API 埋点 |
| **L2 买家可见** | 浏览足迹、收藏夹、最近浏览、个性化推荐 | 基于浏览+询价历史的推荐引擎 |
| **L3 卖家数据参谋** | 访客详情、转化漏斗、热力图、关键词分析 | 实时/离线分析管道 |

### 2.2 阿里卖家侧分析维度（数据参谋）

- **访客画像**：蓝标/金标占比、电商买家类型、访问国家分布、设备类型

- **转化漏斗**：搜索 → 点击 → 浏览详情 → 询盘 → TM 咨询 → 成交
- **2025 Aurora 算法**：多模态意图建模替代关键词匹配，实时分析 session 时长、CTR、跳出率、是否先聊天再询价

### 2.3 阿里买家侧功能

- 浏览足迹（最近浏览商品列表）
- 收藏夹
- 基于浏览 + 询价历史的个性化推荐

## 3. BuildLink 对标策略

### 3.1 分层对标

| 层级 | 阿里能力 | BuildLink MVP（本次） | 后续迭代 |
|---|---|---|---|
| **L1 事件采集** | 全链路前后端埋点 | ✅ 后端核心事件采集（`buyer_events` 表） | 前端 SDK 补充停留时长/滚动深度 |
| **L2 买家可见** | 浏览足迹、收藏、推荐 | ✅ 最近浏览列表 | 收藏夹、个性化推荐 |
| **L3 运营分析** | 数据参谋、漏斗、热力图 | ✅ 基础统计 API（热门商品、漏斗） | 可视化运营面板 |

### 3.2 不做的事（YAGNI）

- ❌ 前端 JS SDK 埋点（停留时长、滚动深度、热力图）
- ❌ 收藏夹功能
- ❌ 个性化推荐引擎
- ❌ 运营侧可视化面板 UI
- ❌ 实时分析管道（Kafka/Flink 等）

## 4. 方案选型

### 4.1 事件采集方式

| 方案 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| **A. 纯后端埋点** | 买方 GET 接口里写 `buyer_events` | 简单可靠、天然有 user_id、不丢数据 | 记不到停留时长等纯前端行为 |
| **B. 纯前端埋点** | 前端 SDK 收集事件批量上报 | 能记停留时长、滚动深度 | 需防刷、需 SDK、复杂度高 |
| **C. 混合** | 后端记转化事件，前端补停留时长 | 覆盖全、灵活 | 两端都要改，复杂度中等 |

**决策：选 A（纯后端埋点）**

理由：
- MVP 核心诉求是转化漏斗（浏览→加购→询价→接受报价），后端 API 调用全覆盖
- "最近浏览"也只需后端记录
- 停留时长等行为分析价值在商品量和用户量起来后才显著，当前 48 个商品不急
- 后续加前端埋点是增量，不影响已有数据

### 4.2 存储方案

| 方案 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| **复用 audit_logs** | GET 事件也写审计表 | 不加表 | 违反 CLAUDE.md（GET 不记审计）、数据量淹没安全审计 |
| **独立 buyer_events 表** | 专表专用 | 职责清晰、可独立清理/归档 | 多一张表 |
| **外部分析服务** | Mixpanel / Amplitude | 功能强大 | 引入外部依赖、数据出境合规 |

**决策：独立 `buyer_events` 表**

理由：
- 与安全审计解耦，互不干扰
- 数据永久保留，作为长期资产（运营分析、数据陈列、AI 客服训练语料）
- 查询模式不同（审计按 user_id + action 查，行为按 event_type + resource_id 聚合）

## 5. 数据治理策略

### 5.1 保留策略

**决策：永久保留，不做定期清理。**

理由：
- 行为数据是长期资产，后续用途包括：
  - 运营分析与数据陈列（转化漏斗、热门商品、买方画像）
  - AI 客服训练语料（按 session 切片还原用户意图路径）
  - 个性化推荐（浏览 + 询价历史驱动）
- 量大后考虑冷热分离（如超过 6 个月的数据归档到分区表），但不删除

### 5.2 表结构设计

```sql
CREATE TABLE buyer_events (
    id              SERIAL PRIMARY KEY,
    buyer_org_id    INT NOT NULL REFERENCES buyer_organizations(id),
    user_id         INT NOT NULL REFERENCES users(id),
    session_id      VARCHAR(36),               -- 会话 ID，前端 sessionStorage 生成
    event_type      VARCHAR(30) NOT NULL,       -- VIEW_PRODUCT / SEARCH / ADD_TO_CART / ...
    resource_type   VARCHAR(30),                -- product / category / rfq / cart
    resource_id     INT,                        -- 对应实体 ID
    referrer        VARCHAR(500),               -- 来源页面，request.headers["referer"]
    device_type     VARCHAR(20),                -- desktop / mobile / tablet，从 UA 解析
    ip              VARCHAR(50),                -- 客户端 IP，地域分析用
    extra           JSONB NOT NULL DEFAULT '{}',-- 搜索词、数量等业务扩展
    created_at      TIMESTAMP NOT NULL
);

CREATE INDEX ix_buyer_events_user_type_time ON buyer_events (user_id, event_type, created_at DESC);
CREATE INDEX ix_buyer_events_resource ON buyer_events (resource_type, resource_id);
CREATE INDEX ix_buyer_events_session ON buyer_events (session_id);
```

**字段取值说明：**

| 字段 | 数据来源 | 备注 |
|---|---|---|
| `session_id` | 前端 `sessionStorage` 生成 UUID，通过 `X-Session-Id` header 传递 | 标签页关闭自动清除 = 天然 session 边界 |
| `referrer` | `request.headers.get("referer")` | 零成本，记录用户从哪个页面触发的请求 |
| `device_type` | 从 `User-Agent` header 解析 | 东非市场手机占比高，区分设备行为差异 |
| `ip` | `request.client.host` | 后续地域分析（国家/城市维度） |
| `extra` | 按 event_type 不同填充 | SEARCH: `{"keyword":"cement","results":12}`；ADD_TO_CART: `{"quantity":100}` |

**事件类型清单：**

| event_type | 触发端点 | resource_type | extra |
|---|---|---|---|
| `VIEW_PRODUCT` | `GET /products/{id}` | product | `{}` |
| `SEARCH` | `GET /products?keyword=xxx` | — | `{"keyword":"...","results":N}` |
| `VIEW_CATEGORY` | `GET /products?category_code=xxx` | category | `{"category_code":"TILES"}` |
| `ADD_TO_CART` | `POST /cart/items` | product | `{"quantity":100}` |
| `CREATE_RFQ` | `POST /rfqs` | rfq | `{"item_count":3}` |
| `SUBMIT_RFQ` | `PATCH /rfqs/{id}/submit` | rfq | `{}` |
| `ACCEPT_QUOTE` | 报价接受端点 | rfq | `{"quote_id":5}` |

**去重策略：** 同一 `user_id` + `event_type` + `resource_id`，5 分钟内不重复写入（防刷新刷量）。

**淘汰字段及理由：**

| 字段 | 为什么不要 |
|---|---|
| ~~`page_url`~~ | 后端模式下就是 API 路径，`event_type + resource_id` 已表达，冗余 |
| ~~`user_agent`~~ | 原始 UA 字符串，`device_type` 已够用，需要时可从 nginx access log 补 |
